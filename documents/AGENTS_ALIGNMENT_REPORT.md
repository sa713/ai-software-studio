# AGENTS_ALIGNMENT_REPORT.md
v0.1
2026-06-19

## Назначение

Данный отчёт фиксирует выравнивание `AGENTS.md` с детальными инструкциями канонических агентов AI Software Studio.

Проверены:

- `documents/AGENTS.md`
- `documents/agents/STUDIO_DIRECTOR.md`
- `documents/agents/PRODUCT_ANALYST.md`
- `documents/agents/PRODUCT_OWNER.md`
- `documents/agents/SOLUTION_ARCHITECT.md`
- `documents/agents/DELIVERY_PLANNER.md`
- `documents/agents/IMPLEMENTER.md`
- `documents/agents/VALIDATOR.md`
- `documents/agents/HISTORIAN.md`
- `documents/agents/RELEASE_MANAGER.md`

# Найденные расхождения

## 1. Product Analyst был описан слишком широко

В старой версии `AGENTS.md` Product Analyst был описан как агент, который "преобразует идеи в требования" и может готовить аналитические разделы PRD.

Это создавало риск смешения Discovery и Product Definition.

Фактическая модель:

- Product Analyst владеет Discovery;
- Product Analyst создаёт Discovery Report;
- Product Analyst не создаёт PRD;
- Product Analyst не владеет требованиями продукта;
- финальное описание продукта и PRD принадлежат Product Owner.

## 2. Product Owner был описан недостаточно явно

Старая версия фиксировала Product Owner как представителя интересов заказчика, но недостаточно явно отражала владение продуктом внутри Студии.

Фактическая модель:

- Product Owner владеет Product Definition;
- Product Owner создаёт Idea Brief и PRD;
- PRD является источником истины для требований продукта;
- Product Owner не занимается архитектурой, планированием, реализацией и тестированием.

## 3. Studio Director был описан без полной границы владения

Старая версия упоминала Project State, но не фиксировала достаточно явно, что Studio Director не создаёт артефакты других ролей.

Фактическая модель:

- Studio Director владеет Project State;
- Studio Director принимает процессные решения;
- Studio Director маршрутизирует работу к владельцам ролей;
- Studio Director не пишет PRD, архитектуру, backlog, код, Validation Report, Release Package или Project Memory.

## 4. Solution Architect был описан слишком кратко

Старая версия перечисляла архитектуру и технологии, но не фиксировала полный контур технических решений.

Фактическая модель:

- Solution Architect владеет Architecture Document;
- Solution Architect самостоятельно принимает технические решения;
- Product Owner владеет продуктом, а Delivery Planner владеет планом реализации;
- Solution Architect не меняет состав продукта и не создаёт Backlog или Task Specifications.

## 5. Delivery Planner был описан недостаточно полно

Старая версия фиксировала backlog, но не отражала полную ответственность за Task Specifications и границы планирования.

Фактическая модель:

- Delivery Planner владеет Backlog и Task Specifications;
- Backlog является источником истины для состава работ;
- Task Specification является контрактом реализации конкретной задачи;
- Delivery Planner не меняет продукт, архитектуру и технологии.

## 6. Implementer был описан без полной границы принятия работы

Старая версия корректно указывала код и тесты, но недостаточно явно отделяла реализацию от независимой проверки.

Фактическая модель:

- Implementer создаёт код, тесты, обновлённую техническую документацию при необходимости и Implementation Report;
- Implementer принимает только локальные реализационные решения внутри утверждённых границ;
- Implementer не принимает собственную работу и не выполняет роль Validator.

## 7. Validator требовал более точной формулировки независимости

Старая версия говорила, что Validator не может проверять работу, которую создал сам.

Фактическая модель требует более широкой формулировки:

- Validator владеет независимой проверкой;
- Validator создаёт Validation Report;
- Validator формирует Validation Status;
- Validator не исправляет код, не меняет требования или архитектуру и не принимает релиз;
- агент не может принимать собственную работу.

## 8. Historian был описан слишком общо

Старая версия фиксировала поддержку памяти проекта, но не отражала границы Project Memory.

Фактическая модель:

- Historian владеет Project Memory;
- Project Memory является источником истины для накопленных знаний проекта;
- Historian фиксирует уже принятые решения, но не принимает решения за другие роли;
- Historian не исправляет артефакты других агентов и не выбирает сторону в существенных противоречиях.

## 9. Release Manager был описан без связи с подтверждённой Validation

Старая версия описывала подготовку релиза, но не фиксировала, что Release Manager оформляет релиз на основании подтверждённых результатов Validation и Release Approval.

Фактическая модель:

- Release Manager владеет Release Package;
- Release Package включает Release Notes и является источником истины для состояния релиза;
- Release Manager не исправляет дефекты, не повторяет Validation, не принимает релиз за заказчика и не обновляет Project Memory.

## 10. Правило минимального состава могло противоречить полной модели ролей

Старая версия говорила, что часть агентов может быть объединена или отключена в небольших проектах.

Это могло создать впечатление, что канонические роли или этапы жизненного цикла можно исключать.

Фактическая модель:

- канонический список ролей сохраняется;
- один исполнитель может совмещать совместимые роли;
- совмещение не отменяет ответственность роли;
- совмещение не должно нарушать независимую проверку.

# Внесённые изменения

`AGENTS.md` обновлён как краткий обзор ролей.

Для каждой канонической роли добавлены или уточнены:

- назначение;
- зона ответственности;
- создаваемые или принадлежащие роли результаты;
- ограничения;
- права принятия решений.

Уточнены общие правила:

- взаимодействие через артефакты, Project Memory и процессные решения Studio Director;
- назначение канонических ролей;
- ответственность специализированных исполнителей;
- независимая проверка;
- эскалация через Studio Director;
- совмещение ролей без изменения канонического списка ролей.

Из `AGENTS.md` не переносились:

- детальные методики работы;
- чеклисты;
- форматы артефактов;
- подробные правила эскалации;
- Definition of Done отдельных инструкций.

# Принятые решения

## 1. AGENTS.md остаётся обзорным документом

`AGENTS.md` должен отвечать на вопрос: "Какие роли существуют в Студии и за что каждая отвечает?"

Операционные детали остаются в `documents/agents/`.

## 2. Владение PRD закреплено только за Product Owner

Product Analyst больше не описывается как владелец требований или участник создания PRD.

Discovery Report остаётся входом для Product Definition, но не заменяет PRD.

## 3. Project State и Project Memory разведены

Project State принадлежит Studio Director и является служебным состоянием процесса.

Project Memory принадлежит Historian и является долговременной памятью проекта.

## 4. Validation отделена от Implementation и Release

Implementer готовит результат к проверке.

Validator независимо проверяет результат и формирует Validation Report.

Release Manager оформляет релиз, но не принимает качество вместо Validator и не утверждает релиз за заказчика.

## 5. Совмещение ролей не создаёт новых ролей

Специализированный исполнитель или подагент может выполнять работу канонической роли, но не становится новой канонической ролью.

Ответственность остаётся за назначенной канонической ролью.

# Дополнительная проверка

## Полномочия без владельца

Не обнаружены.

Каждая ключевая область имеет владельца:

- процесс и Project State — Studio Director;
- Discovery и Discovery Report — Product Analyst;
- продуктовые требования и PRD — Product Owner;
- архитектура и Architecture Document — Solution Architect;
- план реализации, Backlog и Task Specifications — Delivery Planner;
- реализация и Implementation Report — Implementer;
- независимая проверка и Validation Report — Validator;
- долговременная память и Project Memory — Historian;
- релиз и Release Package — Release Manager.

## Владельцы без полномочий

Не обнаружены.

Для каждой роли в `AGENTS.md` зафиксированы права принятия решений внутри её зоны ответственности.

## Пересечение ответственности

Критических пересечений не обнаружено.

Границы уточнены:

- Product Analyst исследует проблему, Product Owner определяет продукт;
- Solution Architect проектирует техническое решение, Delivery Planner планирует реализацию;
- Implementer реализует задачу, Validator проверяет результат;
- Release Manager оформляет релиз, Historian сохраняет память проекта;
- Studio Director управляет процессом, но не подменяет специализированные роли.

## Противоречия между ролями

Нерешённых противоречий не обнаружено.

Обнаруженные расхождения были устранены обновлением `AGENTS.md`.
