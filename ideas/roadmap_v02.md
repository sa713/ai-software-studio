AI Software Studio v0.2 — Экспертная Студия

Статус

Версия 0.1 считается завершённой.

В версии 0.1 построен полный производственный контур:

* Idea
* Discovery
* Product Definition
* Solution Design
* Planning
* Implementation
* Validation
* Release Approval
* Release
* Project Memory Update

Студия умеет превращать идею в работающий программный продукт.

Главное ограничение версии 0.1:

Студия хорошо отвечает на вопрос:

«Как реализовать продукт?»

Но недостаточно хорошо отвечает на вопросы:

* «Правильный ли продукт мы собираемся создавать?»
* «Что заказчик забыл предусмотреть?»
* «Насколько хороша существующая реализация?»
* «Какие проблемы уже есть в существующем проекте?»

Версия 0.2 направлена на усиление экспертности Студии.

⸻

Цель версии 0.2

Добавить способность:

* выполнять полноценный бизнес-анализ;
* находить отсутствующие требования и сценарии;
* критиковать решения и предлагать улучшения;
* анализировать существующие проекты;
* выполнять архитектурное и кодовое ревью;
* работать не только с новыми проектами, но и с уже существующими системами.

⸻

Направление 1. Усиление Product Analyst

Проблема

Текущий Product Analyst хорошо понимает проблему заказчика.

Но его возможности ограничены анализом явно предоставленной информации.

Он недостаточно активно ищет:

* пропущенные требования;
* забытые пользовательские роли;
* отсутствующие сценарии;
* операционные процессы;
* пограничные случаи;
* метрики успеха;
* эксплуатационные риски.

Цель

Перевести Product Analyst из режима Discovery в режим Business Analysis.

Результат

Product Analyst должен не только понимать проблему, но и активно искать пробелы в будущем продукте.

После усиления он обязан выявлять:

* отсутствующие пользовательские сценарии;
* отсутствующие роли пользователей;
* отсутствующие административные процессы;
* отсутствующие механизмы сопровождения;
* отсутствующие механизмы поддержки;
* отсутствующие механизмы мониторинга;
* риски внедрения;
* риски масштабирования;
* риски эксплуатации.

Ожидаемый эффект

Студия начинает предлагать улучшения продукта ещё до появления PRD.

⸻

Направление 2. Усиление Validator

Проблема

Текущий Validator проверяет:

* соответствие требованиям;
* соответствие архитектуре;
* соответствие Acceptance Criteria.

Но почти не оценивает качество самого решения.

Цель

Добавить экспертную оценку результата.

Новые режимы проверки

Compliance Review

Проверка соответствия.

Вопрос:

«Сделано ли то, что было заказано?»

Expert Review

Проверка качества.

Вопрос:

«Насколько хорошим получилось решение?»

Области экспертной оценки

* архитектура;
* качество кода;
* сопровождаемость;
* сложность решения;
* безопасность;
* UX;
* технический долг;
* масштабируемость;
* тестируемость.

Ожидаемый эффект

Validator начинает не только искать ошибки, но и улучшать качество продукта.

⸻

Направление 3. Existing Project Review

Проблема

Студия ориентирована на разработку новых проектов.

Она плохо работает с уже существующими системами.

Цель

Добавить второй режим работы Студии.

Текущий режим:

Idea
↓
Development
↓
Release

Новый режим:

Existing Project
↓
Audit
↓
Review
↓
Improvement Proposal

Поддерживаемые объекты

* Telegram-боты;
* мобильные приложения;
* веб-приложения;
* AI-агенты;
* внутренние инструменты;
* существующие GitHub-репозитории.

Результат

Студия должна уметь принимать на вход:

* исходный код;
* документацию;
* репозиторий;
* архитектурные документы;

и выдавать:

* список проблем;
* список рисков;
* список улучшений;
* план развития.

⸻

Направление 4. Code Review Framework

Проблема

Validator умеет проверять результат задачи.

Но не умеет проводить полноценное ревью существующего проекта.

Цель

Создать единый стандарт технического аудита.

Обязательные виды ревью

Architecture Review

Проверка архитектурных решений.

Code Quality Review

Проверка качества кода.

Security Review

Проверка безопасности.

Maintainability Review

Проверка сопровождаемости.

Dependency Review

Проверка зависимостей.

Test Coverage Review

Проверка тестов и покрытия.

Результат

Студия должна уметь выпускать единый отчёт по качеству существующего проекта.

⸻

Направление 5. Возможный Project Auditor

Статус

Не реализовывать заранее.

Причина

Неизвестно, действительно ли нужна отдельная роль.

Возможно, задачи аудита смогут покрыть:

* Product Analyst;
* Solution Architect;
* Validator.

Условие появления роли

Project Auditor может быть создан только после нескольких реальных аудитов проектов.

Создание роли допускается только через процесс из STUDIO_IMPROVEMENT_PROCESS.md.

⸻

Направление 6. Studio Self-Improvement

Цель

Научить Студию использовать накопленный опыт для улучшения собственной работы.

Примеры вопросов

* Какие возвраты происходят чаще всего?
* Какие требования чаще всего оказываются неполными?
* Какие вопросы чаще всего задаются заказчику?
* Какие решения чаще всего пересматриваются?
* Какие проблемы повторяются от проекта к проекту?
* Какие роли чаще всего становятся источником возвратов?

Источник данных

* Project Memory;
* результаты аудитов;
* Studio Improvement Issues.

Результат

Студия начинает улучшать собственные процессы на основании реальных проектов.

Критерий завершения версии 0.2

Версия 0.2 считается завершённой, если Студия умеет:

* проектировать новые продукты;
* анализировать существующие продукты;
* выполнять бизнес-анализ;
* выполнять архитектурное ревью;
* выполнять код-ревью;
* формировать план улучшения существующих систем;
* использовать накопленный опыт для улучшения собственной работы.
