Mission Mode v03

Цель

Добавить в AI Software Studio второй режим работы поверх существующего Task Mode.

Mission Mode предназначен для автономного достижения крупных целей без необходимости вручную выдавать каждую следующую задачу.

Mission Mode не заменяет Task Mode.

Mission Mode использует существующие роли, процессы и review-gates Студии.

⸻

Проблема

Текущая модель Студии работает по схеме:

Source Of Work → Task → Implementation → Validation

Она хорошо подходит для отдельных задач.

Но для крупных инициатив пользователь вынужден вручную запускать каждый следующий шаг.

Пример:

* проанализировать проект;
* создать roadmap;
* выбрать следующую задачу;
* передать задачу Codex;
* проверить результат;
* выбрать следующую задачу;
* повторять цикл десятки раз.

Mission Mode должен позволять Студии самостоятельно выполнять последовательность задач до достижения заранее определённой цели.

⸻

Основные принципы

Mission Mode не заменяет Task Mode

Task Mode остаётся основным режимом работы.

Mission Mode является надстройкой над ним.

Все реализации продолжают выполняться через обычные задачи.

⸻

Mission не является Task

Task — единица работы.

Mission — источник множества связанных задач.

Одна Mission может содержать десятки Task.

⸻

Mission не даёт права менять продукт

Mission Mode не даёт права:

* менять vision;
* менять цели продукта;
* расширять roadmap;
* придумывать крупные функции;
* принимать продуктовые решения.

Если требуется продуктовое решение, Mission должна остановиться.

⸻

Новая сущность: Mission

Mission содержит:

* Mission ID
* название
* цель
* описание
* ограничения
* входные артефакты
* критерии завершения
* условия остановки
* бюджет автономности
* текущее состояние

⸻

Жизненный цикл Mission

1. Mission Intake

Пользователь создаёт Mission.

Вход:

* репозиторий
* roadmap
* review
* ограничения

Выход:

* Mission Definition

⸻

2. Mission Planning

Studio Director анализирует входные данные.

Определяются:

* scope
* milestones
* ограничения
* бюджет автономности

Выход:

* Mission Plan

⸻

3. Backlog Generation

Delivery Planner создаёт начальный backlog.

Все задачи должны ссылаться на Mission как Source Of Work.

Пример:

Source Type: Mission

Source Reference: MISSION-IDLE-MATH-STAGE-1

⸻

4. Implementation Loop

Для каждой задачи:

* Delivery Planner формулирует задачу
* Implementer выполняет задачу
* Validator проводит проверку
* Historian обновляет знания проекта

После завершения задача закрывается.

⸻

5. Mission Review

После каждого цикла выполняется обзор Mission.

Проверяется:

* что сделано
* какие риски появились
* какие задачи остались
* есть ли блокеры
* достигнут ли milestone

⸻

6. Mission Complete

Mission завершается если:

* достигнут целевой milestone;
* выполнены критерии завершения;
* дальнейшая работа требует новой Mission.

⸻

Роли в Mission Mode

Studio Director

Отвечает за:

* запуск Mission;
* управление жизненным циклом Mission;
* контроль ограничений;
* остановку Mission при необходимости.

⸻

Product Analyst

Отвечает за:

* анализ состояния проекта;
* анализ roadmap;
* выявление рисков;
* выявление возможностей.

Не создаёт задачи напрямую.

⸻

Product Owner

Отвечает за:

* продуктовые решения;
* изменение roadmap;
* принятие новых возможностей.

⸻

Delivery Planner

Отвечает за:

* формирование backlog;
* выбор следующей задачи;
* формулирование задач для Codex.

Не создаёт инициативы.

Все задачи должны иметь Source Of Work.

⸻

Implementer

Отвечает за реализацию задач.

⸻

Validator

Отвечает за приёмку результатов.

Может создавать новые Source Of Work только через замечания и rework.

⸻

Historian

Поддерживает актуальность проектных артефактов.

⸻

Source Of Work в Mission Mode

Task может ссылаться на Mission как источник.

Пример:

Source Type: Mission

Source Reference: MISSION-IDLE-MATH-STAGE-1

Trace:

Mission → Roadmap Stage 1 → Backlog Item → Task

⸻

Budget автономности

Каждая Mission должна иметь ограничения.

Примеры:

* максимум 10 задач подряд;
* максимум 5 циклов;
* максимум 50 изменённых файлов;
* максимум 3 дня работы без участия пользователя.

После достижения лимита Mission обязана остановиться и подготовить отчёт.

⸻

Stop Conditions

Mission обязана остановиться если:

* требуется продуктовое решение;
* найден архитектурный блокер;
* roadmap противоречит проекту;
* есть несколько равноценных направлений развития;
* достигнут milestone;
* исчерпан бюджет автономности.

⸻

Scope Protection

Mission не имеет права:

* расширять roadmap;
* менять vision;
* добавлять крупные функции вне roadmap;
* изменять продуктовую стратегию.

Новые идеи фиксируются как Opportunity.

Opportunity требует решения Product Owner.

⸻

Mission Artifacts

Добавить новые артефакты:

* MISSION.md
* MISSION_STATE.md
* MISSION_BACKLOG.md
* MISSION_REVIEW.md

⸻

Pilot Mission

Первая пилотная Mission:

Mission ID: MISSION-IDLE-MATH-STAGE-1

Goal:

Завершить Stage 1 из IDLE_MATH_LAB_REVIEW_AND_ROADMAP.md

В рамках Mission допускается:

* создание задач;
* реализация задач;
* обновление backlog;
* проведение validation.

Mission должна останавливаться только при:

* завершении Stage 1;
* обнаружении блокера;
* необходимости продуктового решения;
* исчерпании бюджета автономности.

⸻

Критерии успеха Mission Mode

Mission Mode считается успешным если:

* работает поверх существующей архитектуры Студии;
* не ломает Task Mode;
* сохраняет Source Of Work;
* сохраняет review-gates;
* позволяет выполнять несколько задач подряд без участия пользователя;
* не допускает бесконтрольного роста scope;
* успешно проходит пилот на Idle Math Lab Stage 1.
