# CODE_REVIEW_FRAMEWORK.md
v0.2
2026-06-22

## Назначение

Code Review Framework — канонический стандарт технической оценки качества программного продукта внутри AI Software Studio.

Framework определяет обязательные направления проверки, типы находок и правила оценки.

Framework применяется:

- к новым проектам;
- к существующим проектам;
- к Telegram-ботам;
- к мобильным приложениям;
- к веб-приложениям;
- к AI-агентам;
- к внутренним инструментам;
- к существующим GitHub-репозиториям.

Framework не является новой ролью, новым этапом жизненного цикла, новым режимом работы Студии или заменой существующих артефактов.

Framework используется как общий набор критериев для Expert Review, технического ревью и аудитов существующих проектов.

---

# Границы Framework

Framework отвечает на вопрос:

"Насколько качественно устроено программное решение и какие технические проблемы, риски или улучшения нужно зафиксировать?"

Framework не отвечает на вопросы:

- какой продукт нужно создать;
- какие требования должны войти в PRD;
- какую архитектуру нужно выбрать с нуля;
- можно ли принять релиз за заказчика;
- какой этап жизненного цикла должен быть следующим.

Product Owner остаётся владельцем PRD.

Solution Architect остаётся владельцем архитектурного решения.

Validator остаётся владельцем независимой проверки качества на этапе VALIDATION.

Studio Director остаётся владельцем процессного маршрута и решений о возвратах.

Framework не создаёт роль Reviewer, не создаёт роль Project Auditor и не меняет жизненный цикл.

# Обязательные направления проверки

Каждое техническое ревью должно покрывать шесть направлений.

Глубина проверки зависит от Validation Scope, размера проекта, доступных артефактов и риска изменений. Framework задаёт направления и критерии, но не требует механического чеклиста из сотен пунктов.

## 1. Architecture Review

Цель: оценить, насколько структура системы поддерживает требования, развитие, эксплуатацию и дальнейшую реализацию.

Проверить:

- структуру системы;
- границы компонентов;
- зависимости между компонентами;
- связность и сцепление;
- масштабируемость;
- архитектурный долг.

Типовые вопросы:

- Понятно ли, из каких частей состоит система?
- Есть ли у компонентов ясные зоны ответственности?
- Нет ли циклических, неявных или чрезмерно сильных зависимостей?
- Не нарушает ли реализация Architecture Document или принятые архитектурные решения?
- Можно ли развивать систему без непропорционально дорогих изменений?
- Есть ли архитектурные компромиссы, которые нужно явно зафиксировать?

Недопустимо:

- заменять архитектуру только потому, что существует более привычный или любимый подход;
- требовать перепроектирование без доказанного влияния на требования, риски, сопровождение, масштабирование или эксплуатацию;
- подменять Solution Architect задним числом.

## 2. Code Quality Review

Цель: оценить качество кода как рабочего материала для понимания, изменения, проверки и дальнейшего развития.

Проверить:

- читаемость;
- сложность;
- дублирование;
- структуру кода;
- качество абстракций;
- качество ошибок и обработки исключений.

Типовые вопросы:

- Можно ли понять поведение кода без скрытого устного контекста?
- Не является ли решение сложнее задачи?
- Нет ли дублирования, которое создаёт риск расхождения логики?
- Соответствует ли код локальным паттернам проекта?
- Помогают ли абстракции уменьшить сложность, а не прячут её?
- Ошибки обрабатываются явно, диагностируемо и безопасно?

Недопустимо:

- требовать переписывание кода ради эстетики;
- навязывать личный стиль, если текущий стиль не ухудшает сопровождение;
- считать любое отличие от предпочтений Validator дефектом.

## 3. Security Review

Цель: выявить очевидные и practically relevant security risks в пределах технического ревью.

Проверить:

- очевидные уязвимости;
- управление доступом;
- секреты;
- хранение данных;
- интеграции;
- безопасные настройки по умолчанию.

Типовые вопросы:

- Нет ли секретов, токенов, ключей или паролей в коде, конфигурации или документации?
- Есть ли контроль доступа там, где данные или действия требуют ограничения?
- Не раскрываются ли чувствительные данные через логи, ошибки, API или UI?
- Безопасны ли настройки по умолчанию?
- Есть ли риск небезопасной интеграции с внешним сервисом?
- Не создаёт ли решение очевидный путь обхода правил доступа?

Security Review в этом framework не является полноценным pentest, threat modeling exercise или формальной security certification.

Если нужна глубокая проверка безопасности, Validator должен зафиксировать ограничение проверки и риск, а Studio Director определяет дальнейший маршрут.

## 4. Maintainability Review

Цель: оценить, насколько систему можно сопровождать, исправлять и расширять без непропорциональных затрат.

Проверить:

- простоту сопровождения;
- локализацию изменений;
- связанность модулей;
- понятность структуры проекта;
- документацию.

Типовые вопросы:

- Можно ли локально изменить поведение без каскада правок по всему проекту?
- Понятно ли, где искать основные части логики?
- Не смешаны ли пользовательская логика, интеграции, хранение данных и инфраструктурные детали без необходимости?
- Есть ли документация для запуска, настройки, эксплуатации или важных решений?
- Зафиксированы ли известные ограничения и технический долг?

## 5. Dependency Review

Цель: оценить риски внешних зависимостей и их влияние на поддержку, безопасность и развитие проекта.

Проверить:

- внешние зависимости;
- заброшенные библиотеки;
- избыточные зависимости;
- риски обновления.

Типовые вопросы:

- Каждая внешняя зависимость имеет понятное назначение?
- Нет ли зависимости ради маленькой задачи, которую проще решить существующими средствами проекта?
- Не выглядит ли библиотека заброшенной или несовместимой с текущим стеком?
- Есть ли зависимости с очевидными security или maintenance risks?
- Понятны ли последствия обновления ключевых библиотек?

Dependency Review не требует немедленного обновления каждой устаревшей библиотеки. Результат должен отличать реальный риск от обычной версии, которая просто не самая новая.

## 6. Test Coverage Review

Цель: оценить, насколько тесты помогают подтвердить корректность, снизить риск регрессий и поддерживать развитие продукта.

Проверить:

- наличие тестов;
- критические непокрытые области;
- качество тестов;
- стратегию тестирования.

Типовые вопросы:

- Покрыты ли критические пользовательские, интеграционные или бизнес-сценарии?
- Есть ли тесты на изменённую функциональность?
- Проверяют ли тесты поведение, а не только детали реализации?
- Не проходят ли тесты независимо от фактической корректности продукта?
- Есть ли критические negative и edge cases без проверки?
- Можно ли повторно выполнить тесты понятным способом?

Отсутствие полного покрытия не является дефектом само по себе. Дефектом или риском становится отсутствие тестов для критичного поведения, Acceptance Criteria, безопасности, данных, интеграции или области с высокой вероятностью регрессий.

# Типы находок

Framework использует единую модель findings:

- Defect;
- Risk;
- Improvement;
- Subjective Preference.

Каждая находка должна иметь тип, серьёзность, evidence, последствия и рекомендацию.

## Defect

Defect — проверяемое нарушение требования, Acceptance Criteria, Architecture Document, Scope, безопасности, корректности, стабильности, тестируемости, сопровождаемости или другой обязательной границы.

Последствия:

- может блокировать Validation Verdict;
- требует исправления, возврата или решения ответственной роли;
- должен иметь evidence и, если применимо, воспроизводимый сценарий.

Влияние на Validation Verdict:

- Critical Defect всегда ведёт к Failed;
- Major Defect обычно ведёт к Failed, если нет явного обоснования непрепятствующего влияния;
- Minor Defect может вести к Passed With Risks или Required Follow-Up.

Обязательное исправление:

- требуется, если дефект блокирует требование, безопасность, критичный сценарий, архитектурное ограничение или приёмку;
- может быть отложено только явным процессным решением Studio Director.

## Risk

Risk — проблема или неопределённость, которая ещё не является доказанным дефектом, но может повлиять на эксплуатацию, масштабирование, безопасность, сопровождение, стоимость изменений, качество пользовательского результата или будущую разработку.

Последствия:

- может повлиять на Validation Verdict;
- требует осознанного решения, mitigation или follow-up;
- должен объяснять вероятное влияние и область риска.

Влияние на Validation Verdict:

- существенный риск может вести к Failed, если делает результат непригодным для следующего этапа;
- контролируемый риск может вести к Passed With Risks;
- низкий риск может быть non-blocking finding.

Обязательное исправление:

- требуется не всегда;
- обязательно только если риск неприемлем для следующего этапа или нарушает утверждённые ограничения.

## Improvement

Improvement — возможность улучшить качество, простоту, UX, тестируемость, сопровождаемость, безопасность, масштабируемость или эксплуатацию, когда текущее решение соответствует обязательным требованиям и не создаёт существенного риска.

Последствия:

- не является дефектом;
- не должна блокировать Validation сама по себе;
- может стать задачей развития, если Studio Director или ответственная роль сочтёт её полезной.

Влияние на Validation Verdict:

- не ведёт к Failed само по себе;
- может быть отражена в Passed With Risks только если улучшение связано с явным остаточным риском.

Обязательное исправление:

- не требуется без отдельного решения Studio Director.

## Subjective Preference

Subjective Preference — вкусовое, стилевое или альтернативное предпочтение без доказанного влияния на требования, риски, качество, сопровождение или эксплуатацию.

Последствия:

- не является дефектом;
- не является риском;
- не должно превращаться в обязательный follow-up.

Влияние на Validation Verdict:

- не влияет на Validation Verdict;
- может быть записано только как non-blocking observation, если помогает команде понять контекст.

Обязательное исправление:

- не требуется.

# Серьёзность и блокирующий статус

Серьёзность описывает влияние находки.

Blocking status описывает, препятствует ли находка переходу к следующему процессному решению.

Минимальные уровни серьёзности:

- Critical;
- Major;
- Minor;
- Observation.

Critical означает, что проблема блокирует ключевое требование, создаёт высокий security или data risk, ломает критичный сценарий, делает результат непригодным к эксплуатации или делает проверку невозможной.

Major означает, что проблема нарушает важное требование, архитектурное ограничение, значимый сценарий, безопасность, масштабируемость или сопровождаемость, но не обязательно делает весь продукт непригодным.

Minor означает локальную проблему, которую нужно исправить или учесть, но она не блокирует ключевой сценарий.

Observation означает non-blocking finding, который важен для понимания качества, риска или будущего развития.

Improvement и Subjective Preference не должны быть blocking сами по себе.

# Правила оценки

Каждая находка должна отвечать на вопросы:

- что обнаружено;
- почему это важно;
- какие последствия;
- насколько серьёзна проблема;
- что рекомендуется сделать.

Каждая находка должна иметь evidence.

Допустимые источники evidence:

- код;
- тесты;
- результаты выполнения тестов;
- конфигурация;
- документация;
- PRD;
- Architecture Document;
- Task Specification;
- Implementation Report;
- Validation Report;
- Project Memory;
- наблюдаемое поведение системы;
- явно отсутствующая информация, если её отсутствие влияет на оценку.

Запрещается:

- переписывать проект ради красоты;
- навязывать личные предпочтения;
- заменять архитектуру без доказанной причины;
- критиковать без объяснения последствий;
- выдавать Improvement за Defect;
- выдавать Subjective Preference за Risk;
- требовать новое требование через code review;
- использовать Framework как замену PRD, Architecture Document или Validation Report.

Рекомендация должна описывать направление исправления или follow-up, но не обязана содержать полное техническое решение.

Если рекомендация требует продуктового решения, она маршрутизируется к Product Owner через Studio Director.

Если рекомендация требует архитектурного решения, она маршрутизируется к Solution Architect через Studio Director.

Если рекомендация является локальным дефектом реализации, она маршрутизируется к Implementer через Studio Director.

# Использование Framework в Validation

Validator использует Framework как источник критериев для Expert Review и технической части независимой проверки качества.

Во время Validation:

- Compliance Review проверяет соответствие требованиям, PRD, Acceptance Criteria, Architecture Document и Scope;
- Expert Review использует направления Code Review Framework для оценки качества решения;
- Findings классифицируются как Defect, Risk, Improvement или Subjective Preference;
- Improvement Opportunities отделяются от дефектов;
- Validation Verdict объясняет, какие findings blocking, а какие non-blocking.

Framework не даёт Validator право менять требования, перепроектировать архитектуру, принимать релиз или подменять Solution Architect.

---

# Использование Framework для существующих проектов

Framework пригоден для аудита существующих проектов, потому что не зависит от конкретного жизненного цикла одной задачи.

При проверке существующего проекта Framework может использоваться для оценки:

- текущего состояния архитектуры;
- качества кода;
- security risks;
- сопровождаемости;
- зависимостей;
- тестового покрытия;
- технического долга;
- возможностей улучшения.

Режим Existing Project Review, его входы, выходы и маршрут определены в `EXISTING_PROJECT_REVIEW.md`.

# Definition of Done для технического ревью по Framework

Ревью по Framework считается достаточным, если:

- покрыты все шесть обязательных направлений проверки;
- для неприменимых направлений указано `Not Applicable` с причиной;
- каждая находка имеет тип;
- дефекты отделены от рисков;
- улучшения отделены от обязательных исправлений;
- субъективные предпочтения не используются как блокеры;
- каждая значимая находка имеет evidence;
- каждая рекомендация объясняет последствия и маршрут follow-up;
- результат можно использовать в Validation Report или Existing Project Review Report без устного контекста.
